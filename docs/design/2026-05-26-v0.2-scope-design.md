# para-vault-mcp v0.2 — Scope Design

**Date:** 2026-05-26
**Status:** Approved
**Predecessor:** v0.1.0 (shipped 2026-05-18, lab note published 2026-05-25)

## Identity

**v0.2 is the release where para-vault-mcp stops being Robert-shaped.**

A v0.1 user with a vault that doesn't match the maintainer's exact folder and daily-note-section conventions can install the tool and use it. Correctness fixes ride along so the tool's outputs are trustworthy under the broader inputs it will now see. Distribution (npm publish) ships so adoption is a paste-a-command operation, not a clone-and-build operation.

## In Scope

### Foundation

- **#2 — Configurable section names and folder paths** via a config file at `_system/PARA-conventions.md` (frontmatter-based). Defines the keys the tools will read at startup:
  - **Section names** (3): `capture-section`, `work-log-section`, `end-of-day-check-section`.
  - **Folder paths** (3): `daily-notes-folder`, `inbox-folder`, `projects-folder`. These are the only folder paths v0.1 tools actually touch; the other PARA folders (Areas, Resources, Archive) are not read or written and stay out of the schema.
  - Defaults preserved — a vault without this file behaves exactly as v0.1.

### Correctness

- **#1 — `parseFrontmatter` surfaces YAML errors** rather than silently returning empty data. A `_project.md` with malformed frontmatter produces an explicit error (or a `_frontmatterError` field on the parsed note) instead of a phantom "no frontmatter" reading.
- **#3 — `appendToSection` is safe under concurrent writes.** Two parallel `capture` (or `log_work`) calls against the same daily note both land. Mechanism choice (mtime retry, file lock, in-process mutex) deferred to the implementation plan.

### Substrate adjustment

- **#8 — `capture` inserts at top of the section's primary bullet list**, rather than at section end. Removes the manual workaround the maintainer has adopted as the Captures section accumulates LLM-emerged structure (subsections, callout children, strikethrough state).

### Adoption

- **#5 — Publish to npm** as `@robertwtucker/para-vault-mcp`. Lowers install friction from clone-build-register to a single `npx` registration. Strict downstream of #2 — see issue body for the "npx + hardcoded assumptions = footgun" reasoning.

## Out of Scope (with reasoning)

- **#6 — Submit to MCP directories.** Strictly downstream of #5, and a first-impression event with one-time conversion rates. Better to let v0.2 marinate for at least a few weeks of self-use (and ideally one external user's confirmation) before listing.
- **#7 — Structured `get_captures_today` read tool.** The LLM is already doing the interpretive work satisfactorily without a deterministic structured-read tool. Real and architecturally interesting, but not unblocking — deserves its own focused release later.

## Sequencing

```
#2 (configurable conventions)
  └── #1, #3, #8 (independent, parallelizable)
        └── #5 (npm publish — last)
```

- **#2 lands first.** It defines the config-loading plumbing the rest of v0.2 inherits — file location, load timing, default behavior on missing/malformed config. Other items written against the v0.1 hardcoded constants would require rewriting against the v0.2 config plumbing one issue later.
- **#1, #3, and #8 can land in any order, including in parallel.** They touch different files: `src/vault/frontmatter.ts` (#1), `src/vault/daily.ts` (#3), `src/tools/capture.ts` + new helper in `src/vault/daily.ts` (#8).
- **#5 lands last.** npm publishes are observable forever; the configurable surface and bug fixes must be stable before they become external API.

## Done Criteria

v0.2 ships when all of the following are true:

1. **Configurability proof.** A vault with non-default section names (e.g., `## Inbox` instead of `## Captures`) and/or non-default folder paths (e.g., `Projects/` instead of `1-Projects/`) declared in `_system/PARA-conventions.md` works end-to-end across all five tools without code changes.
2. **Default preservation.** A vault with no `_system/PARA-conventions.md` keeps working with v0.1 defaults. Existing test fixtures (which use defaults) stay green.
3. **#1 verified.** The original repro frontmatter (unquoted `#` in `next-action`) produces a surfaced error or `_frontmatterError` field — not silent empty data.
4. **#3 verified.** A test exercising parallel `capture` or `log_work` calls against the same daily note demonstrates no lost updates.
5. **#8 verified.** A `capture` into a Captures section containing subsections lands above the first subsection, in the primary bullet list, at the top.
6. **`npx -y @robertwtucker/para-vault-mcp` starts cleanly** against a real vault from the published tarball — not just from the local repo.
7. **README updated** to lead with the `npx` install path and document `_system/PARA-conventions.md`. CHANGELOG `[Unreleased]` promoted to `[0.2.0]` with date.
8. **No open `v0.2` milestone issues remain.**

## Documentation Impact

- **README:** Install section rewritten to lead with `npx`, with clone-build retained as the contributor-development path. "Vault expectations" section gains a paragraph on `_system/PARA-conventions.md`. The "first for the maintainer's own vault" framing is no longer the headline.
- **CHANGELOG:** `[Unreleased]` accumulates entries as work lands. At ship time, promote to `[0.2.0]` with that day's date, prose-led (consistent with v0.1.0).
- **CONTRIBUTING.md:** Brief note on testing against a non-default vault, since the v0.2 substrate makes that meaningful for the first time.
- **No new top-level docs.** The "tools provide substrate, LLMs interpret convention" principle is worth stating somewhere eventually, but that essay belongs in a later release (likely tied to #7), not v0.2.

## Resolved Decisions

- **npm scope:** `@robertwtucker/para-vault-mcp` (scoped, not unscoped).
- **Config file location:** Hardcoded as `_system/PARA-conventions.md`. The config file path is the one convention that stays hardcoded in v0.2 — accept the trade-off. v0.3 can add an env-var override (`PARA_CONFIG_PATH`) if anyone asks.
- **Config schema scope:** v0.2 declares *exactly* six keys — three section names (`capture-section`, `work-log-section`, `end-of-day-check-section`) and three folder paths (`daily-notes-folder`, `inbox-folder`, `projects-folder`). Insertion behavior, callout markers, sub-section conventions, date format, etc. do **not** ship as config keys in v0.2, even if discussed. Future config keys require their own issue and release — treat the config schema as semver-versioned API.
- **#8 ships as default change, no config option.** The semantic argument (capture is inbox, not log) generalizes; the maintainer's manual workaround is empirical evidence. If a future user requests append-bottom, push back before adding a config knob.
- **#7 (`get_captures_today`) deferred on its own merits**, not because of #8. They are architecturally adjacent (write substrate vs. read substrate) but independent.

## Deferred to Implementation Plan

- **Concurrent-write mechanism choice for #3** (mtime retry vs. `proper-lockfile` vs. in-process mutex). The spec records the constraint (no lost updates); the plan picks the mechanism.
- **`_frontmatterError` field shape vs. stderr logging for #1.** Issue body proposes both options; final choice is a plan-level call.
- **Test layout for non-default vaults.** Whether the test suite adds parallel fixtures with non-default names, or parameterizes existing fixtures, is a plan decision.
- **Section-heading matching details for #2.** Heading level (assume `##` matching v0.1, or accept any level), case sensitivity (default to case-sensitive — markdown headings are case-sensitive in render), and missing-section behavior (preserve v0.1's create-if-missing semantics). None of these warrant config knobs; they are plan-level resolutions of how the lookup works in practice.

## Risks Accepted

- The hardcoded config file path means a user who renames `_system/` cannot use the tool. Acceptable for v0.2; revisit if it actually comes up.
- npm publishes are functionally permanent within 72 hours. Done criteria #6 (verify from published tarball) and the sequencing rule (#5 last) mitigate this.
- License headers (SPDX) on new files must not slip. Mechanical, but worth a plan-level checklist.
- **Plugin-generated sections are unsupported.** v0.2 operates on static markdown sections. If a user maps a tool to a section whose content is plugin-generated (e.g., `tasks`, `dataview`, or other fenced code blocks owned by an Obsidian plugin), behavior is undefined and out of scope. Detecting or warning about such cases is a v0.3+ consideration; for v0.2 the contract is "we operate on static markdown."
