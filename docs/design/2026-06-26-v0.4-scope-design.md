# para-vault-mcp v0.4 — Scope Design

**Date:** 2026-06-26
**Status:** Draft
**Predecessor:** v0.3.0 (shipped 2026-06-14, lab note published 2026-06-17 — *The review composed the tool*)

## Identity

**v0.4 is the release where failures fail loudly.**

v0.3 framed the discipline as the review composing the tool — the tool tells you when it can't answer, refuses parameters that would encode private conventions, narrows its surface to what every PARA practitioner would recognize. v0.4 extends that discipline downward, into every silent path the v0.3 ship left behind.

A v0.3 user passing a typoed date string gets `[]` and no error. A user with `area: [[Engineering]]` written unquoted gets `[]` and no error. A user with wikilink aliases (`[[Areas/Health|Health]]`) gets `[]` and no error. The CI configuration audits as safe because no scanner was checking it. The cold-path opener does three serialized I/O round-trips that no one was timing.

None of these are crashes. None will be reported as bugs. They are **silences** — the tool returning a confident-looking answer that happens to be wrong, or wasting time without telling you. v0.4 makes each one either correct or loud.

## In Scope

### Security hardening

The two issues surfaced by enabling CodeQL static analysis on 2026-06-26. Both are workflow-config issues, not application bugs — but they're exactly the class of silence v0.4 wants to end: the CI was scored as safe because nothing was scoring it.

- **#22 — `ci.yml` test job lacks an explicit `permissions:` block.** Inherits the default broad `GITHUB_TOKEN`. `release.yml` already declares an explicit block; this is a one-file gap. Add `permissions: contents: read` to the test job — read-only is sufficient for typecheck + vitest.
- **#23 — `pnpm/action-setup@v6` is referenced by moving tag rather than pinned commit SHA** in both `ci.yml` and `release.yml`. Pin to a commit SHA with a trailing `# v6` comment. `release.yml`'s `id-token: write` for OIDC trusted-publish makes it the higher-blast-radius case. Dependabot's `github-actions` ecosystem (already enabled) will keep SHAs current.

### find_project correctness

The spine of the release: four bugs in `src/vault/projects.ts` where the tool returns wrong or empty answers for valid Obsidian YAML shapes and frontmatter typos.

- **#15 — Bare unquoted `area: [[Engineering]]` is parsed by js-yaml as a nested array and silently dropped to `undefined`** by `loadProject`'s `typeof === 'string'` check. Project never matches `area:` filters. Extend area parsing to recognize the single-element-nested-array shape and collapse it through `normalizeArea` like every other wikilink presentation.
- **#16 — `normalizeArea` doesn't handle wikilink aliases of the form `[[Areas/Health|Health]]`.** Yields `"areas/health|health"` instead of `"health"`, so the alias literally written by the user fails to match. Parse the wikilink as `[[target(|alias)?]]`, prefer the alias when present, and fall back to the last path segment of the target. All four canonical shapes — `[[Health]]`, `[[Areas/Health]]`, `[[Areas/Health|Health]]`, `"[[Areas/Health|Health]]"` — collapse to the same canonical form.
- **#14 — `toDateString` silently shifts dates when YAML carries an explicit timezone offset.** `value.toISOString().slice(0, 10)` is correct for date-only YAML (parsed to UTC midnight) but wrong for timestamps-with-offset, which it slices in UTC and shifts by up to a day. Affects `due`, `updated`, `last-reviewed` for any user writing timestamps from Templater or other generators.
- **#18 — `parseDateString` silently accepts impossible dates (`2026-13-45`) via `new Date()`'s rollover semantics**, producing wrong-but-confident results in filters and frontmatter. Validate that the parsed Date round-trips to the same year/month/day components; otherwise return `undefined`.

These four cluster into two natural sub-PRs by code path: **{#15, #16}** both touch area normalization; **{#14, #18}** both touch date parsing. Each sub-cluster lands as one PR.

### Internal cleanup

- **#20 — `SORT_KEY_MAP` is a four-entry table where three entries are identity mappings.** Adding a sortable field requires touching four parallel places. Drop the camelCase internal alias (`lastReviewed`) in favor of the YAML-honest `last_reviewed` end-to-end, deleting the map. This is cleanup, but it's a precondition for clean perf work in #19 — caching parsed Dates into a renamed field is easier if the rename has already happened.

### Performance

The cold-path waste surfaced by PR #13's review. Neither is a correctness bug; both are silences in the sense that they cost time without telling you.

- **#17 — `inboxStatus` does three independent I/O operations sequentially** (daily-note read, inbox readdir+stat, previous-daily-note readdir). Promise.all them. Roughly 2-3× speedup on the cold path.
- **#19 — `loadProject` parses `data.updated` into a Date for `daysSinceUpdate`, discards it, and the `updated_since` filter then re-parses the same string for every project on every call.** Cache the parsed Date alongside the string form. Rides downstream of #14 and #18 (so the cached Date is correct) and downstream of #20 (so the field naming is clean).

## Out of Scope (with reasoning)

Nothing currently. All nine v0.4 candidates fit the "silences" theme and are small. If post-spec implementation surfaces that any one is unexpectedly large or risky, defer it to v0.5 at that point — but the spec ships with all nine in.

If a tenth correctness or hardening issue is filed during v0.4 implementation, evaluate against the theme: silent-failure-class items belong in v0.4 even if filed late; new-feature requests go to v0.5.

## Sequencing

```
Wave 1 — Security hardening (independent, safe)
  #22 → #23
  Lands first because it costs nothing and the CI all subsequent PRs run through
  is hardened from that point forward.

Wave 2 — Correctness (the spine)
  PR-B: #15 + #16        [normalizeArea / wikilink shapes]
  PR-C: #14 + #18        [parseDateString / toDateString]
  Wave 2 can land in parallel — the two sub-clusters edit overlapping files but
  non-overlapping helpers. If parallel feels risky, land PR-C first (smaller,
  more contained) then PR-B.

Wave 3 — Cleanup that unlocks perf
  #20 (snake_case rename, map collapse)
  Single PR. Precondition for #19's clean caching.

Wave 4 — Performance
  #17 (Promise.all in daily.ts — independent, can land alongside or before #19)
  #19 (cache parsed Date — depends on #14, #18, #20 all already merged)

Wave 5 — Release
  CHANGELOG promote, version bump, tag, ship, lab note.
  Doc-refresh pass: update Zod .describe() strings on find_project's
  parameters and the tool description itself to reflect dateErrors and
  the post-#18 exclusion criteria (invalid `updated:` now also drops
  from stale_days / updated_since, not just missing `updated:`). The
  describes went stale during Wave 2 PR-C; folded here rather than
  filed as a separate v0.4 issue.
```

- **#22/#23 land first.** Defense in depth: every subsequent PR's CI inherits the hardening.
- **The correctness spine lands second.** It's what the release exists for.
- **#20 lands before #19.** #19 stashes the parsed Date into a `ProjectSummary` field whose name #20 changes. Doing #19 against the pre-rename shape and then renaming is two passes through the same code.
- **#19 lands after #14 and #18.** The whole value of caching the parsed Date evaporates if the parse itself is still wrong.
- **#17 is independent.** Land it anywhere in Wave 4.

## Done Criteria

v0.4 ships when all of the following are true:

1. **CodeQL clean.** Subsequent CodeQL runs on `main` surface no `actions/missing-workflow-permissions` or `actions/unpinned-tag` findings against the repo's workflows.
2. **Wikilink shapes verified.** A `_project.md` with `area:` written in each of the four canonical shapes (`[[Health]]`, `[[Areas/Health]]`, `[[Areas/Health|Health]]`, `"[[Areas/Health|Health]]"`, plus bare `Health`) all match a `find_project({area: 'Health'})` call.
3. **Unquoted-wikilink test added.** The test suite includes the case the v0.3 author admitted to missing in #15 — `area: [[Foo]]` bare unquoted.
4. **Impossible-date test added.** `find_project({updated_since: '2026-13-45'})` returns a Zod validation error, not `[]`; frontmatter carrying `updated: 2026-13-45` surfaces a frontmatter-error-style signal (mechanism deferred to plan — see below).
5. **TZ-offset test added.** A `_project.md` with `due: 2026-06-30T20:00:00-08:00` round-trips through `find_project` reporting `due: "2026-06-30"`, not `"2026-07-01"`.
6. **Sort key extension verified.** Adding a hypothetical sort key requires touching at most two places (Zod enum + `sortValue` switch), not four.
7. **Latency probe.** A timed `inboxStatus` call on a cold cache demonstrates the Promise.all path completes in roughly the slowest single-op time, not the sum.
8. **Caching probe.** A `find_project({updated_since: …})` against a fixture with N projects shows N total `parseDateString` calls per request, not 2N.
9. **No open `v0.4` milestone issues remain.**
10. **CHANGELOG `[Unreleased]` promoted to `[0.4.0]`** with date and prose-led entries grouped under the silences theme (not a flat bullet list of issue numbers).

## Resolved Decisions

- **The release identity is "failures fail loudly" / "make the silences loud"**, not "v0.4 — bug fixes and security hardening." A theme that justifies #20 (refactor) and #17 (perf) under the same banner as #15 and #22.
- **All nine candidates are in.** No deferrals at spec time. The theme is broad enough to hold a refactor and a perf win alongside the correctness spine without diluting.
- **Issues #14, #15, #16, #18 are correctness fixes, not features.** They do not require new tool parameters, new config knobs, or new public API. The fix surface lives entirely inside `loadProject`, `normalizeArea`, `parseDateString`, `toDateString`. The [[feedback-tools-vs-conventions|tools-vs-conventions]] principle is upheld — no new convention-encoding parameter shipped.
- **CHANGELOG sub-grouping.** Use the Keep-A-Changelog Security / Changed / Fixed / Performance subheadings rather than a flat bullet list, so the silences theme is visible at a glance.

### Implementation forks resolved (2026-06-26)

The scope doc originally deferred three "proposed approach" forks to the implementation plan. All three resolved during scope review:

- **#14 (TZ shift) — inspect raw YAML for the `T` separator.** `@11ty/gray-matter` exposes the raw YAML block string via the `matter` property on its return value (`GrayMatterFile.matter: string`), so the implementation reads the raw value for a given key from `parsed.matter` and checks for `T` to distinguish date-only from timestamp. Rejected the cheaper "probe parsed Date for UTC midnight" approach because it has a residual silence — a timestamp with an offset that mathematically lands on UTC midnight (e.g. `2026-06-30T16:00:00-08:00` → `2026-07-01T00:00:00Z`) is misclassified as date-only and silently shifts. The edge case is contrived but the theme is "no residual silences"; spending five extra lines to close it is cheap. Second time this session the [[reference-gray-matter-replacement|@11ty/gray-matter swap]] paid off architecturally — the maintained fork's API made this option available without a parser fork.
- **#18 (impossible dates) — surface a per-field signal on `ProjectSummary`.** `parseDateString` returns `undefined` for invalid dates (so they drop from filters and from `daysSinceUpdate`) **and** the parsed project surfaces the failure in find_project results so the LLM can flag the bad value to the user. Rejected the silent-undefined and stderr-only paths: both leave the LLM in the dark on a value that's actively wrong. The exact carrier shape — extending the existing whole-file `frontmatterError` vs. adding a per-field `dateErrors?: {field, value}[]` — remains a plan call; the existing `frontmatterError` is a whole-file error so a new per-field field is the cleaner fit.
- **#19 (Date caching) — stash internal `_<field>Date?: Date` fields on `ProjectSummary`.** Add `_updatedDate`, `_dueDate`, `_lastReviewedDate` as underscore-prefixed internal fields, parsed once at `loadProject` (alongside the #18 round-trip validation), reused by `daysSinceUpdate` and the `updated_since` filter. JSON serialization filters out `_`-prefixed keys at the response boundary. Rejected the "normalize all dates to epoch-ms internally" path: wider blast radius (changes sortValue, every comparator, every string consumer), and the round-trip it claims to kill is already 80% dead once the parsed Date is stashed alongside the string. The fork resolution shifted as #18 locked: caching becomes a natural extension of work `loadProject` is already doing for round-trip validation, not a separate perf concern.

## Deferred to Implementation Plan

- **Test fixture additions.** Wikilink-shape parameterization, TZ-offset frontmatter, and `2026-13-45`-style fixtures are new. Whether to parameterize one fixture or add four discrete ones is plan-level.
- **`dateErrors` carrier shape (#18).** Per-field error field on `ProjectSummary` vs. extending the existing `frontmatterError`. Both work; the per-field field is cleaner because `frontmatterError` is a whole-file shape — the plan locks the exact name and structure.
- **Raw-value extraction strategy (#14).** Whether to slice `parsed.matter` with a per-key regex or to re-parse the YAML block with a position-tracking schema is plan-level. A regex is sufficient for v0.4's flat date fields; a position-tracking parser would generalize to multi-line scalars we don't ship today.
- **CodeQL re-run timing.** Verify #22/#23 clean before merging Wave 2 PRs, so any new finding gets a fast PR-level signal.
- **`pnpm/action-setup@v6` SHA resolution.** Don't bake the SHA into the spec; resolve at PR-author time from the action's tags.

## Risks Accepted

- **The lab note's "silences" framing is post-hoc.** None of the issues were originally filed under the silences theme; the scope doc retro-fits them into one. That's fine for a release identity, but the lab-note risk is that the framing is more compelling to the author than the reader. Mitigation: write the lab note draft against the audience that hit silently-wrong outputs in v0.3, not against the author.
- **#16 (wikilink aliases) is not currently exercised in the maintainer's vault.** All Integration / Engineering / Health-style area values are flat. A regression in `normalizeArea` for the flat case would land in production without dogfooding catching it. Mitigation: PR-B's test suite covers all four canonical shapes; flat is the existing test case, alias-with-display and alias-without-display are the new cases.
- **#19 (Date caching) refactors a hot path.** The perf win is real but small relative to N. A bug here that returns wrong staleness silently is worse than the bug the spec is supposedly closing — a silent miscount of `daysSinceUpdate` is *itself* a silence-class failure. Mitigation: the existing `daysSinceUpdate` tests stay green at every step; #19 is a refactor, not a behavior change.
- **CodeQL findings may regrow.** Adding new third-party actions (e.g., a coverage uploader) re-introduces `actions/unpinned-tag`. The pattern, not just the specific findings, has to hold. Mitigation: lab note frames the pattern as a discipline; CONTRIBUTING.md gets a note about pinning third-party actions.
